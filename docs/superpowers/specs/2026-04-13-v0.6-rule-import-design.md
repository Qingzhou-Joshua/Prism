# v0.6 Rule Import Design

**Date:** 2026-04-13  
**Status:** Approved

## Goal

让用户能在 Scanner 页面直接将各平台现有的 `.md` 规则文件导入到 Prism 统一规则库，补全"扫描 → 导入 → 统一管理 → 发布"的完整闭环。

## Architecture

采用懒加载方案：点击平台卡片上的"查看可导入规则"按钮时，才触发数据请求。后端 `GET /platforms/:id/rules` 接口已在 v0.3 实现，复用 `adapter.importRules()`。前端并发请求平台规则列表与 Prism 现有规则，在客户端做冲突检测，最终复用已有的 `POST /rules` 和 `PUT /rules/:id` 接口写入，无需新增任何 API。

**Tech Stack:** Fastify (backend), React + TypeScript (frontend), Playwright (E2E)

---

## Data Flow

```
Scanner 页面（App.tsx） → 点击"查看可导入规则"
  → 并发:
      GET /platforms/:id/rules  (v0.3 已实现)
        → adapter.importRules()  (v0.3 已实现)
        ← { platformId: string, items: ImportableRule[] }
      GET /rules  (已有)
        ← UnifiedRule[]
  → 前端按 name 字段做冲突检测
  → 渲染规则列表（新建 / 冲突-跳过 / 冲突-覆盖）
  → 用户确认导入
      新规则 → POST /rules
      覆盖规则 → PUT /rules/:id
```


## Backend

### Existing Route (no changes needed)

`GET /platforms/:id/rules` 已在 `packages/server/src/routes/platforms.ts` 实现：

```
Response: { platformId: string, items: ImportableRule[] }
```

- 通过 `registry` 找到对应 adapter
- 调用 `adapter.importRules?.()` — 可选方法
- adapter 不支持 `importRules` 时返回 **404**（而非空列表）
- 平台 id 不存在 → **404**

前端需要将 404 响应视为"该平台不支持规则导入"，展示空状态而非报错。

### No New Write API

导入写入复用已有接口：
- 新建：`POST /rules`
- 覆盖：`PUT /rules/:id`

### Per-Rule State Machine

```
new           → 默认勾选，绿色"新建"标签
conflict-skip → 默认，黄色"冲突"标签 + 跳过下拉
conflict-overwrite → 用户切换，警告色
imported      → 导入成功后标记
failed        → 单条导入失败，红色标记
skipped       → 用户选择跳过或未勾选
```

### Import Flow

1. 点击"导入已选 N 条" → 逐条串行执行
2. 每条完成/失败后更新该条状态
3. 全部完成后显示汇总：`✅ 成功 2 条，⚠️ 跳过 1 条，❌ 失败 0 条`
4. 导入成功的规则状态变为"已导入"，按钮变灰

`apps/web/src/api/platformRules.ts` — 封装 `GET /platforms/:id/rules` 调用，返回类型为 `{ platformId: string, items: ImportableRule[] }`

## Error Handling

| 场景 | 处理 |
|------|------|
| `detected=false` 或 `rulesDetected=false` | 不显示"查看规则"按钮 |
| `GET /platforms/:id/rules` 返回 404 | 视为"不支持导入"，展示空状态，不报错 |
| `GET /platforms/:id/rules` 返回其他错误 | 展开区内联错误 + 重试按钮 |
| 单条导入失败 | 标记失败，继续其余条目，最终汇总 |
| 导入中途收起卡片 | 已完成的保留，未完成的停止（幂等） |
